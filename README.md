# ABC Musical Instruments — Analytical Data Warehouse

A small SQLite data warehouse and ETL pipeline for ABC Musical Instruments'
order data, replacing their previous Excel-based process. Built entirely in
Python (standard library `sqlite3` + `pandas` for data wrangling — no ORM,
no external database engine).

## What this is

- A **star-schema** data model: one fact table (order lines) plus three
  dimension tables (customer, product, date).
- A **modular ETL pipeline**: extract -> clean/validate -> load, with row-level
  error handling so one bad file doesn't stop the whole run.
- **Slowly Changing Dimension (Type 2)** support for customers and products,
  so address/contact changes and price changes are versioned, not overwritten.
- Fully **idempotent and resilient**: delete `warehouse.db` and rerun, or
  rerun the same file twice — you get the same result either way.

## Data model

```
dim_customer                dim_product                dim_date
------------------           ------------------           ------------------
customer_key (PK)            product_key (PK)             date_key (PK)
customer_id                  product_id                   full_date
client_name                  product_name                 year / quarter / month
address / city / postcode    product_type                 day / day_of_week
country / contact_number     unit_price                    month_name / day_name
valid_from / valid_to        valid_from / valid_to         is_weekend
is_current                   is_current
                                       \                         /
                                        \                       /
                                    fact_order_lines
                                    ------------------
                                    order_line_key (PK, surrogate)
                                    order_line_id (natural key, unique)
                                    order_id / line_number
                                    customer_key (FK) / product_key (FK) / date_key (FK)
                                    quantity / unit_price / total_price / currency
                                    payment_type / payment_billing_code
                                    delivery_address / city / postcode / country / contact_number
                                    loaded_at
```

**Why a star schema, and why these tables:** the brief asks for a warehouse
that supports sales reporting and customer/inventory management — a
fact-and-dimension split is the standard shape for that, keeps the table
count small (4 tables), and avoids the redundancy of the original flat CSV
(e.g. the same customer's address repeated on every order line).

**A data quality note on order numbers:** in the source data, the `OrderNumber`
column (e.g. `PO0060504-1`) does **not** reliably group lines into the same
logical order — adjacent line numbers can belong to different clients and
different `PaymentBillingCode`s. So `OrderNumber` is treated as the fact
table's line-level natural key (`order_line_id`), while the true business
order grouping (`order_id`) is derived from `PaymentBillingCode`, which is
consistent per real order.

**Other cleaning applied:**
- Customer name matching is case/punctuation-insensitive (`"Howell LLC"` and
  `"HOWELL LLC"` become one customer), with a canonical display name chosen
  automatically.
- Country names are normalised to a single spelling (`"UK"`, `"UNITED
  KINGDOM"` -> `"United Kingdom"`).
- `TotalPrice` is recomputed as `UnitPrice x Quantity` (trusting the
  derivable value over a possibly-wrong source figure), rather than
  rejecting on a mismatch.

## Setup

Requires Python 3.9+.

```bash
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

## Running the pipeline

Load the bundled sample data (creates/updates `warehouse.db` in the project root):

```bash
python run_pipeline.py
```

Load a specific file (CSV or XLSX):

```bash
python run_pipeline.py --input data/raw/orders_2023.csv
```

Load several files in one run:

```bash
python run_pipeline.py --input data/raw/orders_jan.csv data/raw/orders_feb.csv
```

Fully rebuild the database from scratch (drops and recreates all tables first):

```bash
python run_pipeline.py --reset
```

Use a different database file:

```bash
python run_pipeline.py --db /path/to/other.db
```

**Resiliency check:** you can delete `warehouse.db` entirely and rerun
`python run_pipeline.py` — the pipeline recreates the schema and reloads
everything from the raw CSV. Re-running the same file again afterwards does
not create duplicate rows (facts are upserted by their natural key, and
dimension versions are only added when something actually changed).

## Error handling

Any row that fails validation (missing required field, non-numeric price,
non-positive quantity, invalid date, duplicate order number within the file)
is **not** loaded, but is written out with a reason to
`data/rejects/<filename>_rejects_<timestamp>.csv`, and the rest of the file
still loads normally. A summary is printed to the console, e.g.:

```
[WARNING] data/raw/orders_2023.csv: 3 row(s) rejected, details written to data/rejects/orders_2023_rejects_20260714T120000.csv
[OK] data/raw/orders_2023.csv: 23 row(s) loaded (3 rejected out of 26 read)
```

If a whole file can't be read at all (missing file, unsupported type,
missing required columns), that file is skipped and reported as failed;
other files passed in the same run are still processed.

## Slowly Changing Dimensions

`dim_customer` and `dim_product` are versioned (SCD Type 2): each row has
`valid_from`, `valid_to`, and `is_current`. When a later-dated load shows a
customer's address (or a product's price) has changed, the old row is closed
(`valid_to` set, `is_current = 0`) and a new current row is inserted —
existing fact rows keep pointing at the version that was true at the time.

## Running the tests

```bash
pytest
```

Covers: normalisation/validation logic, the date-dimension builder, and
end-to-end pipeline behaviour (schema creation, idempotent reruns, DB
recreation after deletion, SCD2 versioning, and handling of a file
containing bad rows).
